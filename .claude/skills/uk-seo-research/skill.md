# UK SEO Research для Ultimate.net.ua

## Quick Start

```
/uk-seo-research {slug}
```

**Результат:**
- `uk/categories/{slug}/research/RESEARCH_PROMPT.md` — завантажується цілком в Perplexity
- `uk/categories/{slug}/research/RESEARCH_DATA.md` — скелет для результатів

---

## Workflow

```
/uk-seo-research {slug}
├─► 1. Read _clean.json → keywords, entities, micro_intents, parent_id
├─► 2. Resolve section ID via data/category_ids.json
├─► 3. Find section in PRODUCTS_LIST.md by "## ... (ID: {id})"
├─► 4. Extract product insights (форми/об'єми/ефекти з описів)
├─► 5. Generate RESEARCH_PROMPT.md (структура нижче)
└─► 6. Create RESEARCH_DATA.md skeleton
```

---

## Джерела даних

| Файл | Що беремо |
|------|-----------|
| `uk/categories/{slug}/data/{slug}_clean.json` | keywords, entities, micro_intents, parent_id |
| `data/category_ids.json` | slug → section ID в PRODUCTS_LIST.md |
| `data/generated/PRODUCTS_LIST.md` | Товари секції `## ... (ID: NNN)` |

> ⚠️ **Shared sections:** Кілька slug можуть мапитися на один ID (наприклад antimoshka + antibitum). Фільтруй товари за релевантністю до поточного slug — використовуй назву/призначення товару, не бери все підряд.

## Структура RESEARCH_PROMPT.md

Генеруй файл з цими секціями (в цьому порядку):

### Шапка
- `# {Назва} ({slug}) — SEO Research (Buyer Guide)`
- **ТЗ:** задача + як працювати з файлом (завантажити в Perplexity цілком, відповідь повернути у форматі шаблону)
- **Контекст:** тип сторінки, категорія, parent_id, мова: **українська**

### Семантика (з _clean.json)
- Таблиця `| Ключ | Об'єм |` — всі keywords
- **Entities:** список
- **Питання користувачів:** micro_intents

### Product Insights (з PRODUCTS_LIST.md)

> ℹ️ Інсайти з описів — орієнтир, не гарантія наявності.

| Характеристика | З описів |
|----------------|----------|
| Форми випуску | гель, спрей... |
| Об'єми | 250мл, 500мл... |
| База | водна, силіконова (або "не вказано") |
| Ефекти | матовий, глянцевий (або "не вказано") |
| pH / тип | кислотний, лужний (або "не застосовно") |
| Розведення | 1:5, готовий (або "не вказано") |
| Призначення | шини, пластик (або "не вказано") |

**Якщо товари 2-в-1 або змішаного призначення:**
Додати питання для дослідження: "Чи можна одним складом обробляти X та Y? Чи є обмеження?"

### Промпт для Perplexity (11 блоків)

1. **Що це і навіщо** — визначення, проблема, головне правило
2. **Види/типи** — класифікація за НЕЗАЛЕЖНИМИ осями (див. нижче)
3. **Як обрати** — сценарії "якщо X → бери Y"
4. **Як застосовувати** — підготовка → нанесення → сушка → фініш
5. **Критичні помилки** — помилка | наслідок | як правильно
6. **Безпека** — PPE, обмеження, ризики
6а. **Спірні твердження** — запит на PROOF (див. нижче)
7. **FAQ новачка** — 8-12 коротких Q→A
8. **Troubleshooting** — симптом | причина | рішення
9. **Сумісність** — поверхня | можна? | умови
10. **Практичні цифри** — витрата, час, стійкість, температура (тільки з джерелами)

## Блок 2: Правила класифікації

> ⚠️ **Критично:** Не змішувати осі! Наприклад, силікон — це активний компонент, а не тип носія. Силікон може бути і у водній, і в розчинній базі.

### Типові осі для автохімії/дресингів:

**Вісь 1 — Носій (що випаровується):**
- Water-based (водна)
- Solvent-based (розчинник)

**Вісь 2 — Активний компонент (що залишається):**
- Silicone-based
- Polymer-based
- Natural oils
- (залежить від типу категорії)

**Вісь 3 — Фініш/ефект:**
- Matte (матовий)
- Satin (сатиновий)
- Gloss / Wet look

**Формат у промпті:**
```markdown
> ⚠️ **Важливо:** [Компонент X] — це активний компонент, а не тип носія.
> [Компонент X] може бути і у водній, і в розчинній базі. Не змішувати осі!

**Вісь 1 — Носій:**
| Тип | Опис | Плюси | Мінуси |
|-----|------|-------|--------|

**Вісь 2 — Активний компонент:**
| Тип | Опис | Плюси | Мінуси |
|-----|------|-------|--------|

**Вісь 3 — Фініш:**
| Тип | Опис | Коли обирати |
|-----|------|--------------|
```

Див. [category-matrix.md](../seo-research/references/category-matrix.md) для осей за типами категорій.

## Блок 6а: Спірні твердження

Для кожної категорії виявити типові "міфи" або твердження, що потребують перевірки.

**Формат у промпті:**
```markdown
### Блок 6а: Спірні твердження — запит на PROOF

> ⚠️ Ці твердження часто зустрічаються в контенті, але можуть бути міфами.
> Потрібні **тільки підтверджені джерела** (дослідження, офіційні позиції, тести).

| Твердження | Статус | Джерело/Proof |
|------------|--------|---------------|
| "{спірне твердження 1}" | ? | Потрібне джерело |
| "{спірне твердження 2}" | ? | Потрібне джерело |

**Формат відповіді:** Для кожного:
- "Підтверджено" + URL + цитата
- "Спростовано" + URL + цитата
- "Не знайдено доказів"
```

**Як виявляти спірні твердження:**
1. Прочитати існуючий контент категорії (якщо є)
2. Перевірити entities/micro_intents на потенційно спірні теми
3. Типові патерни:
   - "X шкодить/руйнує Y"
   - "X безпечніше ніж Y"
   - "Не можна використовувати X на Y"
   - "X впливає на Z"

## Вимоги до відповіді (у промпті)

```markdown
## Вимоги до відповідей

1. **Кожен факт = джерело.** Формат: твердження → URL → цитата.
2. **Без джерела — не писати.** Немає даних → "дані не знайдено".
3. **Пріоритет джерел:**
   - Офіційні сайти виробників
   - Наукові статті / тести з вимірюваннями
   - Детейлінг-форуми — тільки для практичних порад
   - YouTube-тести з вимірюваннями
4. **Мова джерел:** англійська, українська, російська.
5. **Актуальність:** бажано 2020-2025.
6. **Спірні твердження:** блок 6а — не підтверджувати без proof.
```
