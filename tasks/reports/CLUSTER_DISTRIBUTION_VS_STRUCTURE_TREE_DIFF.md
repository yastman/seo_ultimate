# Сверка: `CLUSTER_DISTRIBUTION.md` vs `STRUCTURE_TREE.md`

Файлы:
- `tasks/reports/CLUSTER_DISTRIBUTION.md`
- `tasks/reports/STRUCTURE_TREE.md`

## Результат сопоставления

- **Несоответствий по путям (опечатки/неизвестные категории): не найдено.**  
  Все назначения (destination) из `CLUSTER_DISTRIBUTION.md` присутствуют в дереве `STRUCTURE_TREE.md`.

## Найденные расхождения/проблемы

### 1) Узлы структуры без маппинга кластеров (семантика не назначена)

В `STRUCTURE_TREE.md` есть **8** узлов, которые **ни разу не встречаются** как destination в `CLUSTER_DISTRIBUTION.md`:

- `L1: Полировка`
- `L1: Уход за интерьером`
- `L1: Оборудование`
- `L1: Мойка и Экстерьер → L2: Автошампуни`
- `L1: Мойка и Экстерьер → L2: Средства для стекол`
- `L1: Мойка и Экстерьер → L2: Средства для дисков и шин`
- `L1: Аксессуары → L2: Щетки и кисти`
- `L1: Аксессуары → L2: Наборы`

Практический эффект: если генерация/заполнение категорий опирается на `CLUSTER_DISTRIBUTION.md`, эти страницы могут оказаться **без закреплённой семантики** (и/или без контента/метаданных) либо вовсе не попадут в выгрузку.

### 2) Дублирование назначения (две строки на один destination)

Один destination встречается более 1 раза:

- `L1: Полировка → L2: Полировальные круги`  
  (есть и `page`-строка, и `merge`-строка)

Практический эффект: при сборке реестра/выгрузки может потребоваться дедупликация, иначе возможны двойные начисления ключей или конфликты “page vs merge”.

### 3) Уже отмеченные предупреждения по качеству семантики (⚠️) в `CLUSTER_DISTRIBUTION.md`

- `L1: Мойка и Экстерьер → L2: Очистители двигателя` — внутри есть “общая мойка авто”, требуется чистка/перенос.
- `L1: Полировка → L2: Полировальные круги` — в general-кластере есть ключи уровня “оборудование”, возможно перенести в “машинки”.
- `L1: Защитные покрытия → L2: Воски → L3: Жидкий воск` — ключи про “жидкое стекло” перенести в `L2: Керамика и жидкое стекло`.
